02
The MMV mapping problem — why it's hard
Problem Framing ▶
What is MMV mapping?
MMV = Make · Model · Variant. Each insurance provider (Digit, ICICI, Tata AIG, Kotak, etc.) maintains their own internal vehicle master dataset with different naming conventions. To generate an insurance quote, the Vahaan vehicle data must be matched to the correct row in each insurer's master — this is MMV mapping. A wrong match = wrong premium or quote failure.
Why matching is non-trivial
Different naming conventionsVahaan: "MARUTI SUZUKI SWIFT DZIRE VDI" · Insurer: "Swift Dzire VDI (D)" — same vehicle, completely different string representations.
Abbreviations & expansions"BS4" vs "BSIV" vs "Bharat Stage IV" — emission norms written in multiple formats across different insurer masters.
Roman numeralsNumeric variants like "1.2" vs "1200cc", "CNG" vs "CNG Kit", engine sizes expressed differently per insurer.
Partial matchesVahaan data is often truncated or abbreviated. Full variant names in insurer masters may not match any substring directly.
Inconsistent spacingExtra spaces, hyphens, slashes, mixed case — "Swift-Dzire" vs "Swift Dzire" vs "SWIFT DZIRE" all refer to the same vehicle.
Why RDBMS failed firstTried SQL LIKE / ILIKE queries initially — too slow at scale, no ranking of partial matches, couldn't handle fuzzy or keyword-weight based matching.

03
BM25 matching engine — algorithm & design
Core Engineering ▶
What is BM25?
BM25 (Best Match 25) is a probabilistic ranking function from information retrieval — the same algorithm that powers search engines like Elasticsearch and Solr. Given a query (Vahaan vehicle string) and a corpus (insurer master entries), it scores each document by how relevant it is to the query. It improves on simple TF-IDF by saturating term frequency (a word appearing 100 times isn't 100x more useful than appearing 5 times) and normalising for document length (short records shouldn't be penalised vs long ones).
BM25 formula
Score(D, Q) = Σ IDF(qᵢ) · [ f(qᵢ, D) · (k₁ + 1) ] / [ f(qᵢ, D) + k₁ · (1 - b + b · |D|/avgdl) ]

where:
Q = query terms from normalised Vahaan string
D = a candidate entry in insurer MMV master
f(qᵢ, D) = frequency of term qᵢ in document D
IDF(qᵢ) = log( (N - n(qᵢ) + 0.5) / (n(qᵢ) + 0.5) + 1 )
|D| = length of document D (num tokens)
avgdl = average document length across all masters
k₁ ≈ 1.5 = term frequency saturation parameter
b ≈ 0.75 = document length normalisation parameter

→ Higher score = better match between Vahaan string and insurer record
Why BM25 over alternatives
vs SQL LIKE / ILIKE
LIKE requires exact substring presence. No partial credit, no ranking, slow at scale. BM25 ranks all candidates and returns best match.
LIKE — rejected
vs Simple TF-IDF
TF-IDF doesn't saturate term frequency or normalise document length. BM25 handles both — more robust for short MMV strings.
TF-IDF — limited
vs Embedding / semantic search
Keyword-based matching is explainable and fast. Vehicle names are structured identifiers — semantic similarity less meaningful than keyword overlap.
BM25 — chosen

04
String normalisation & preprocessing pipeline
Development ▶
Normalised string construction
A single normalised string was created by combining fields: Make + Model + Variant + Fuel Type + Cubic Capacity (CC) + Gross Vehicle Weight (GVW). This combined string was created from both the Vahaan response and each insurer master row before BM25 matching — ensuring comparison is always string-to-string on the same feature space.
Normalisation steps applied
Lowercasing & whitespaceAll text lowercased, extra spaces collapsed, leading/trailing whitespace stripped, hyphens and slashes standardised.
Abbreviation expansionLookup table for common abbreviations: "CNG" → "compressed natural gas", "PET" → "petrol", "DSL" → "diesel", manufacturer abbreviations expanded.
Roman numeral normalisationBS4 ↔ BSIV, BS6 ↔ BSVI, variant numbers like "1.2L" → "1200cc" — handled via regex + lookup dictionary.
Special character removalBrackets, dots, commas, ampersands stripped or standardised — "(D)" → "diesel", "4WD" → "four wheel drive".
Stopword removalInsurance-domain stopwords removed before indexing — words like "limited", "edition", "series" that appear ubiquitously and carry low discriminative value.
Numeric normalisationCubic capacity values aligned to consistent format (1200, 1500, 998 etc.), GVW aligned to kg units — prevents mismatches from unit inconsistencies.
Per-insurer master normalisation
Each insurer (Digit, ICICI, Tata AIG, Kotak, etc.) has its own vehicle master with unique formatting quirks. Separate normalisation rules were implemented and tested per insurer — the pipeline applies insurer-specific pre-processing before BM25 indexing, so each insurer master is cleaned against its own observed patterns rather than a single global rule set.

08
Production incident — Insure24 outage & resolution
Incident + Learning ▶
⚠ Production outage — Insure24 platform down 4–5 hours
To save infrastructure cost, the MMV mapping service was initially deployed in a shared GitHub repository alongside multiple other rule-based applications. A change to a shared dependency or deployment configuration in that repo caused a conflict — the MMV mapping service broke in production, taking down the Insure24 insurance platform for 4–5 hours. This was a critical business impact incident as users could not get insurance quotes.
Resolution steps (led end-to-end)
Immediate triageIdentified root cause as shared repo deployment conflict — conflicting dependency versions from another application in the same repo.
Worked through the nightSat day and night to isolate, fix, and restore the service — Insure24 was restored within the outage window before business hours.
Repository migrationMoved entire MMV mapping service to a dedicated standalone GitHub repository — complete decoupling from all other applications.
Independent CI/CD pipelineNew GitHub Actions workflow created for the standalone repo — no shared steps, no shared dependencies with other services.
Async refactorDuring the migration, also refactored synchronous calls to async — reduced inference latency as a side benefit of the incident-driven work.
Deployment standardsEstablished the rule: each production service must have its own repository. No shared deployment repos for distinct production services.
Key learning for interview: This incident demonstrated end-to-end ownership — not just building but maintaining, debugging, and improving a production system under pressure. The async refactor that came out of this work also permanently improved the service's inference latency. Cost-cutting shortcuts in infrastructure have non-linear risk when the service is customer-facing.
